Git - 分支学习教程

发表于 2026-03-30 分类于 技术 ， Git

前言

教程地址：

Learn Git Branching

基础片

1.git commit

  • 提交记录

修改代码后，把这些修改保存成一个 提交记录C2

2.git branch

  • 新建分支

看得出来 新建分支新建到了当前所在的记录

通过 git checkout 命令选择到分支，然后把提交记录，这个记录就提交到新的分支里了！

小技巧：通过 git checkout -b 来创建一个新的分支并切换到新的分支

3.git merge

  • 合并分支

通过 git merge 合并

焦点位于main，执行上述命令后，c4新记录将包含c2和c3，且层c4为main分支，这意味main包含代码库的所有最终修改

在通过 git checkout bugFix 和 git merge main 可以让bugFix移动至main，让每一个分支都包含代码库的所有修改

4.git rebase

  • 第二种合并分支

一种更线性的分支合并方式

高级片

前言 ：在提交树上移动

HEAD

HEAD 是当前所在提交记录的符合名称，它表示标识你正在工作的那个提交记录，HEAD总是指向当前分支上最近一次提交记录，通常情况下是指向分支名的

分离HEAD

让HEAD指向某个具体的提交而不是分支

git checkout c1

注意：”c1” 是哈希值

相对引用

在真正的 Git 世界里，哈希值并不是简单的 c1，而是四十位的长文本，你需要通过 git log 命令查看哈希值，Git 对哈希值的处理很聪明，你只需要提供能够唯一标识提交记录的前几个字符即可，而不是那 四十位的长文本！

实际上，我们有更聪明的指定提交记录的方式：相对引用

  • 使用 ^ 向上移动 1 个提交记录

  • 使用 ~ 向上移动 num 个提交记录

  • 修改分支位置

git branch -f main HEAD~3 这条命令会将main强制指向HEAD的第 3 级 parent 提交记录

撤销变更

1. git reset

通过将分支引用向后移动至一个更早的提交来撤销变更，这会把 分支 向上移动，就像之前指向的提交从未发生过一样

例：git reset HEAD^

向上撤销变更一个提交

2. git revert

虽然 git reset 很方便，但是这对 远程分支 是无效的（线上合作）

例子：git revert HEAD

移动工作

“我要这个工作到这里，那个工作到那里”

  • git cherry-pick <提交1> <提交2> <…>

将 提交1 提交2 … 复制到当前所在位置下（HEAD）

  • git rebase -i HEAD~4

将 当前位置 的上 4 条记录整理为对话框，点击按钮取消操作，复制没有取消操作的记录

通过在对话框中拖动记录来整理 提交顺序

  • git rebase -i 与 git cherry-pick

常用于单独将某一提交复制到某分支下

杂项

提交的技巧 #1

当我想修改一个旧提交的一些代码

先用 git rebase -i 重新排序，将我要修改的记录排到前面

然后用 git commit –amend 进行小修改

最后用 git rebase -i 把他们调整回原来顺序

把 main 移动到最前面

提交的技巧 #2

第一种方法会排序两次，可能造成由 rebase 引发的冲突，使用 git cherry-pick 来修改

cherry-pick 可以把提交树上的任何地方的提交记录取过来追加到 HEAD 上！（除了HEAD上游的）

永久标签

实际上，你会发现，分支一直在变，又或者一些临时的分支

所以分支更像开发模式中的标识符

我们需要一个比分支更好的标签能永远指向某一个提交的里程碑的东西：

  • git tags

git tag <哈希值>

让该哈希值指向的提交永久获得 tag-name 的标签，如果 不指定提交记录，tag-name 会指向 HEAD

查找最近标签

  • git describe

git describe

可以是任何能被 git 识别成提交记录 的引用，若没指定，git 会使用 HEAD

返回值：--g

tag 表示离 ref 最近的标签，numCommits 是 ref 与 tag 相差多少个提交记录，hash 表示 ref 表示的提交记录哈希值的前几位

当 ref 本身上就有 标签 时，咋只输出 标签名

高级话题

多分支 rebase

实际上，git rebase的真正用法是：

git rebase

将 your-branch 和它上面的所有没有分支的提交记录 复制到 to-branch

两个 parent 节点

在之前的 git merge 中，你会注意到，合并的分支有两个父提交，这时候你如果用 ^ 操作符，你会发现只会回到第一个父提交上，这是 git 默认的

  • 操作符的链式操作

当然你也可以在任何HEAD上游创建分支

例：git branch newBranch HEAD^2

纠缠不清的分支（总结）

简单总结出现过的 Git 命令

基础操作

  1. git commit
    • 将修改保存为提交记录。
  2. git branch <branch-name>
    • 新建分支（基于当前提交记录）。
  3. git checkout <branch-name>
    • 切换到指定分支。
    • git checkout -b <branch-name>：新建并切换到新分支。
  4. git merge <branch-name>
    • 合并指定分支到当前分支（非线性历史）。
  5. git rebase <branch-name>
    • 将当前分支的提交“变基”到目标分支（线性历史）。

高级操作

  1. HEAD 操作
    • git checkout <commit-hash>：分离 HEAD，指向具体提交（而非分支）。
    • 相对引用：
      • ^：向上移动 1 个提交（如 HEAD^）。
      • ~<num>：向上移动 num 个提交（如 HEAD~3）。
    • 强制移动分支：
      • git branch -f <branch-name> HEAD~<num>：将分支强制指向某提交。
  2. 撤销变更
    • git reset HEAD^：本地撤销提交（移动分支引用，丢弃后续提交）。
    • git revert HEAD：生成反向提交（安全撤销，适用于远程分支）。
  3. 移动提交
    • git cherry-pick <commit1> <commit2>：复制指定提交到当前分支。
    • 交互式变基：
      • git rebase -i HEAD~<num>：整理提交（排序、合并、修改等）。

提交与标签

  1. 修改旧提交
    • 通过 git rebase -i 重新排序提交，用 git commit --amend 修改内容，再调整回原顺序。
    • 或直接用 git cherry-pick 复制修改后的提交。
  2. 标签（Tag）
    • git tag <tag-name> <commit-hash>：为提交打永久标签（未指定哈希时默认指向 HEAD）。
    • git describe <ref>：显示最近标签信息（格式：<tag>-<numCommits>-g<hash>）。

多分支与复杂操作

  1. 多分支变基
    • git rebase <to-branch> <your-branch>：将 your-branch 及其提交复制到 to-branch。
  2. 多父提交处理
    • 合并后的提交有两个父节点，默认 ^ 指向第一个父节点。
    • 链式操作：如 HEAD^2~ 表示第一个父节点的第二个父节点的上一个提交。
    • 从任意位置创建分支：如 git branch newBranch HEAD~^2~。

总结表

命令 用途
git commit 保存修改为提交
git branch 新建分支
git checkout 切换分支/提交
git merge 合并分支（非线性）
git rebase 变基合并（线性历史）
git reset 本地撤销提交
git revert 生成反向提交（安全撤销）
git cherry-pick 复制指定提交
git rebase -i 交互式整理提交
git tag 打永久标签
git describe 显示最近标签信息

远程仓库的 Push & Pull

  • 何为远程仓库？

远程仓库并不复杂, 在如今的云计算盛行的世界很容易把远程仓库想象成一个富有魔力的东西, 但实际上它们只是你的仓库在另个一台计算机上的拷贝。你可以通过因特网与这台计算机通信 —— 也就是增加或是获取提交记录

创建远程仓库

  • git clone

在真实环境下作用：在 本地 创建一个 远程仓库 的拷贝

在教程环境下作用：在 远程 创建一个 本地仓库 的副本

远程分支

你会发现 本地仓库 中多了一个 o/main 的分支，这便是 远程分支 ，他会反应远程仓库的状态

远程分支命名规范：/

所以 o/main 是一个仓库名为 o 中的 main 分支

从远程仓库获取数据

  • git fetch

他会从远程仓库下载本地仓库缺失的 提交记录

更新远程分支指针（如o/main），将所有远程分支更新为远程仓库中相应分支最新的状态

注意！这是一个单纯的下载操作，并不会改变你本地仓库的数据和状态

将获取的远程数据更新到我们的工作中

  • git pull

其实就是 git fetch 和 git merge 的缩写，先下载，后合并

模拟团队合作

由于教程并不能真正的在世界上某个地方改变远程仓库，所以教程自建了一个命令用来模拟在远程仓库上的一个提交

上传我的本地内容

  • git push

不带参数时，默认值与你github版本有关

历史偏移

实际上，当你周一克隆了远程仓库，然后开始研发新功能，但是到周五你发现你的同事写了一堆代码，还改了你功能中使用的api，这些会变动会导致你新开发的功能不可用，但是他们已经将那些提交推送到远程仓库了，因此你的工作就变成了基于项目旧版的代码，与远程仓库最新的代码不匹配了

因为这情况（历史偏离）有许多的不确定性，Git 是不会允许你 push 变更的。实际上它会强制你先合并远程最新的代码，然后才能分享你的工作。

  • 办法 1：下载最新分支，移动你的更改至最新分支下，提交

  • 办法 2：下载最新分支，合并你的更改，提交

  • 办法 3：–rebase参数

  • 办法 4：直接使用 git pull

见图 办法2

远程服务器拒绝！（Remote Rejected）

如果你在大的合作团队中工作，很可能是main被锁定了, 需要一些 Pull Request 流程来合并修改。如果你直接提交(commit)到本地main, 然后试图推送(push)修改, 你将会收到这样类似的信息:

？为什么会被拒绝

远程服务器拒绝直接推送(push)提交到main, 因为策略配置要求 pull requests 来提交更新.

你应该按照流程,新建一个分支, 推送(push)这个分支并申请pull request,但是你忘记并直接提交给了main.现在你卡住并且无法推送你的更新.
